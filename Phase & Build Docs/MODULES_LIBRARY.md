# Baseplate Module Library
## Reusable Infrastructure Modules for AI-Built B2B SaaS

> **🔧 MAINTENANCE:** The structural inventory below ("Live Inventory") is **generated from source** — run `pnpm sync-modules` (`scripts/sync-module-inventory.mjs`) after adding or changing any package/module/migration. The narrative descriptions in "Module Inventory" are human-maintained; add a prose entry whenever you add a new module. For current phase status, see [`../MASTER_PROGRESS.md`](../MASTER_PROGRESS.md).

**Purpose:** Central registry and documentation for all reusable modules that AI agents can use to scaffold B2B applications

**Vision:** When a developer (or AI agent) says "build me a HIPAA clinic portal," Baseplate's module library provides:
- ✅ Authentication (works everywhere)
- ✅ RBAC (works everywhere)
- ✅ Audit logging (required for compliance)
- ✅ Payment processing (works everywhere)
- ✅ Notifications (works everywhere)

Instead of reinventing these for each vertical, **reuse the library.**

## Module Inventory (Phase 1 Build Plan)

### Core Modules (Phase 1A-1B)

#### `auth`
**What it does:** Email/password login + magic link auth  
**Use case:** Every B2B app needs authentication  
**Vertical-specific parts:** None (fully reusable)  
**Implementation:** Supabase Auth wrapper  
**Status:** Built in Phase 1  

**Usage Across Verticals:**
- Med Spa portal: Clinic owner + staff + patient login
- Home Services portal: Dispatcher + technician + customer login
- Contractor portal: Project manager + crew + client login
- **Same module, different roles**

---

#### `rbac`
**What it does:** Check user permissions ("Can this user view this patient?")  
**Use case:** Every B2B app has different user roles with different access levels  
**Vertical-specific parts:** Role names are vertical-specific, but permission system is not  
**Implementation:** Role + permission checking utilities  
**Status:** Built in Phase 1  

**Usage Across Verticals:**
- Med Spa: owner (all), staff (their patients), patient (self)
- Home Services: owner (all), dispatcher (jobs), technician (assigned jobs), customer (self)
- Contractor: owner (all), manager (jobs), crew (assigned), client (self)
- **Same permission system, different roles**

---

#### `audit-logs`
**What it does:** Log every action for compliance ("Who viewed Patient X? When? From where?")  
**Use case:** HIPAA + compliance requirements  
**Vertical-specific parts:** None (same audit trail works for all)  
**Implementation:** Database logging + query utilities  
**Status:** Built in Phase 1  

**Usage Across Verticals:**
- Med Spa: "Dr. Smith viewed Patient Sarah's intake on 6/14 at 2:15pm from 192.168.1.1"
- Home Services: "Dispatcher assigned Job #42 to Tech John on 6/14 at 2:15pm"
- Contractor: "Manager approved invoice from Subcontractor ABC on 6/14"
- **Same audit mechanism, different entities**

---

#### `encryption`
**What it does:** Encrypt sensitive data (payments, medical records, documents)  
**Use case:** HIPAA + data security requirements  
**Vertical-specific parts:** None (encryption algorithm works everywhere)

**Status:** Built in Phase 1

**Use case:** Development, staging, production configurations

#### `monitoring`
**What it does:** Structured leveled logging (`logError`, `logInfo`, `logWarn`, `logMetric`) with automatic secret redaction and JSON (prod) / colorized (dev) output  
**Use case:** Observability for every server-side entry point — error boundaries, API routes, webhooks, background jobs  
**Vertical-specific parts:** None — logging is universal  
**Implementation:** Zero-dependency `process.stdout` emitter; redacts keys matching `key/token/secret/password/auth`  
**Status:** Built in Phase 5 — 4 exports  

---

**Status:** Built in Phase 5 — 7 exports (`getSubscription`, `getSubscriptionStatus`, `updateSubscriptionStatus`, `hasActiveSubscription`, `createCheckoutSession`, `createBillingPortalSession`, `getStripeWebhookEvent`)

## Module Development Guidelines

### When Writing a Module

**1. Ask: "Will this module be used by multiple verticals?"**
- YES → Put in `packages/`
- NO → Put in `apps/portal-[vertical]/`

**2. Ask: "Does this module have vertical-specific code?"**
- YES → Extract vertical parts into config
- NO → Module is ready for reuse

**3. Ask: "Is this a pattern other verticals might use?"**
- YES → Put in `packages/patterns/`
- NO → Put in `packages/core/` or `packages/integrations/`

### Module Template

```typescript
// packages/[category]/[module-name]/index.ts

/**
 * Module: [Name]
 * 
 * Purpose: [What it does]
 * 
 * Use case: [When you'd use this]
 * 
 * Vertical-specific parts: [None / Component X / Config Y]
 * 
 * Example:
 * ```typescript
 * import { [FunctionName] } from '@baseplate/[category]/[module]';
 * 
 * const result = await [FunctionName]({ /* params */ });
 * ```
 */

export async function moduleName(params: Params): Promise<Result> {
  // Implementation
}

export interface Params {
  // Define params
}

export interface Result {
  // Define result
}
```

## Module Governance

### Naming Convention
- Folder: kebab-case (`digital-signature`, `audit-logs`)
- File: kebab-case (`index.ts`, `types.ts`, `utils.ts`)
- Exports: camelCase (`auditLog`, `digitalSignature`)
- Types: PascalCase (`AuditLogParams`, `SignatureResult`)

### Versioning
- Each major module gets version (v1, v2)
- Backwards compatible within major version
- Phase 2: All modules at v1
- Phase 3: Some modules at v2 (with deprecation warnings)

### Testing
- Every module must have tests
- Test coverage >80% before open-source
- Integration tests for modules that work together

### Documentation
- Every module has README.md
- Every exported function has JSDoc comment
- Every type is documented
